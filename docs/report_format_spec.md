# セキュリティ実装レポート 出力仕様書

このファイルは、reporter.pyがレポートを生成する際に
厳守すべきフォーマット仕様を定義したものである。

サンプルファイル（report_sample_v3.txt）と必ずセットで参照すること。
サンプルは「実際にどう見えるか」の見本であり、
この仕様書は「なぜそうなっているか・何が固定で何が可変か」を
説明するものである。


## 1. 出力形式

- 拡張子：.txt（.mdではない）
- 文字コード：UTF-8
- Markdownの太字記号（**）やリスト記号（- や *）は一切使用しない
  理由：メモ帳などのプレーンテキストエディタで開いた際に、
  記号がそのまま表示されて読みにくくなるため


## 2. 全体構成（固定）

レポートは以下の順序で必ずこの4セクションを出力する。
該当項目が0件のセクションも、見出しだけは省略せず出力する
（「該当項目はありません」のような文言を1行入れる）。

1. 自動実装済み
2. 要手動対応（実装はあるが確認が必要）
3. 未実装（対象ファイルが見つかりませんでした）
4. 追加で検討推奨（このツールの対象外）


## 3. ヘッダー部分（固定）

ファイル冒頭は以下の3行で固定する。

セキュリティ実装レポート
生成日時: [YYYY-MM-DD HH:MM:SS]
対象プロジェクト: [対象フォルダ名]

この3行の後は2行の空行を空けてからセクション区切り線に進む。


## 4. セクション見出しのフォーマット（固定）

以下の3行構成で統一する。

==================================================
[セクション名]
==================================================

- 区切り文字は半角イコール(=)を50個
- セクション名に太字記号などの装飾は付けない
  （区切り線自体が見出しとしての強調の役割を果たす）
- セクション見出しの前後にはそれぞれ2行の空行を入れる


## 5. 各項目のフォーマット（固定）

1セクション内の各項目は以下の構成で統一する。

[項目番号] 項目名

対象ファイル: [ファイル名]　※またはステータス: [状態]
　　（自動実装済みセクションでは「対象ファイル」、
　　　既存実装が問題ない場合は「状態: 実装済みのため変更なし」
　　　のように使い分ける。詳細は6項を参照）

内容:
[説明文]

- 項目番号は[1][2][3]のように半角角括弧を使う
- 項目名は項目番号と同じ行に書く
- 項目見出しの後は1行空けてから「対象ファイル」または「状態」行
- その後1行空けてから「内容:」、さらに1行空けてから本文
- 項目と項目の間は区切り線（半角ハイフンを50個）を入れ、
  区切り線の前後にそれぞれ2行の空行を入れる
- セクション内の最後の項目の後には区切り線を入れない
  （次のセクション見出しの区切り線と連続させない）


## 6. セクションごとの記載ルール

### 6-1. 自動実装済み

このツールが実際にコードを書き換えた項目、または
既に実装済みで変更不要だった項目を記載する。

- コードを書き換えた場合：「対象ファイル: [ファイル名]」
- 既に実装済みだった場合：「状態: 実装済みのため変更なし」

内容には「何が問題だったか」「何を追加・確認したか」
「それによってどんな効果があるか」を含めることが望ましいが、
これは分かりやすさのための推奨であり、必須の3点セットではない。
実際の判定結果に応じて、AIが適切と判断した説明を記載してよい。

### 6-2. 要手動対応

実装はあるが、AIが「不完全」「要確認」と判定した項目を記載する。

- 「対象ファイル: [ファイル名]」を記載する
- 「現状の問題:」として、何が・なぜ問題なのかを具体的に書く
- 「推奨する対応:」として、どう直すべきかの方向性を書く
- 最後に「→ [ファイル名] を確認してください」を1行追加する

現状の問題・推奨する対応の文章量に上限はないが、
1文が長くなりすぎる場合は「。」の後で改行を入れて
読みやすさを優先する。

### 6-3. 未実装

対象ファイル自体が見つからず、判定もできなかった項目を記載する。

- 「内容:」のみを記載し、「対象ファイル」「状態」行は省略する
- なぜ見つからなかったか、見つかればどう判断する予定だったかを
  簡潔に説明する

### 6-4. 追加で検討推奨

このツールの対象外だが、ユーザーが知っておくとよい項目を記載する。
この項目は固定の4項目（HTTPS強制・ログ監視・依存パッケージの
定期スキャン・リバースプロキシ運用）を基本とするが、
将来的に項目が追加・変更される可能性がある。
その場合もこのセクションのフォーマット（[番号] 項目名 → 内容:）
は維持すること。


## 7. 改行・余白に関する全体ルール

- 1行あたりの文字数が長くなりすぎないよう、
  「。」（句点）の後で適宜改行する
  目安：1行20〜25文字程度を超えたら次の文の前で改行を検討する
- セクションとセクションの間、項目と項目の間は
  必ず空行を使って余白を作り、文字が詰まった印象にしない
- ファイルの末尾（最後のセクションの最後の項目の後）には
  2行の空行を入れて終える


## 8. 注意事項

- サンプルファイル（report_sample_v3.txt）の中身の文章表現
  （「bcryptを使ったハッシュ化処理を追加しました」等）は、
  あくまで「このように具体的にわかりやすく書く」という
  見本であり、実際の生成時にこの文言をそのまま使う
  必要はない。判定結果に応じて適切な文章を生成すること。
- 項目名・ファイル名・説明文の中身は対象プロジェクトの
  実際のスキャン結果に応じて毎回変わるが、
  上記1〜7のフォーマットルール（区切り線・余白・見出し構成）
  は常に固定として厳守すること。
